## Tool Use Protocol Guardrail

This project is often edited through opencode with local Qwen-family coding models. Tool calls must be emitted only through the runtime's native tool-call channel.

Hard rules:

1. Never print, quote, simulate, or manually write tool calls in assistant text.
2. Never output XML-style pseudo tool invocations, including angle-bracket tags named `tool_call`, `function=...`, or `parameter=...`.
3. Never output JSON-style pseudo tool calls in markdown or plain text.
4. When a file must be changed, use the actual opencode edit/write tool provided by the runtime. The tool invocation must not appear in the visible answer.
5. If the runtime tool channel is unavailable, do not fake a tool call. Instead, stop and say: `TOOL_CALL_UNAVAILABLE`, then provide a normal unified diff or concise manual patch instructions.
6. Do not wrap tool parameters in markdown fences, XML tags, or explanatory prose.
7. Do not emit `<thinking>` or `</thinking>` tags. Keep reasoning private and only summarize decisions briefly when needed.
8. For edits, first read the target file when necessary, then use exact old/new replacement strings. The `oldString` must match the current file content exactly.
9. After editing, run the smallest relevant verification command when available. If verification cannot be run, state that explicitly.
10. Visible responses should describe completed changes and verification results only. They must not contain raw tool syntax.


本文件用于约束本项目中的 AI Agent、自动化编码工具、opencode 代理和人工协作者的开发行为。  
所有自动化修改、构建、测试、下载、临时文件生成和日志输出都必须遵守本文规则。

重要提示：遇到感觉自己无法解决的问题，再多次循环或尝试无结果后应及时询问用户处理方案，不得假装解决或用hardcode、特殊规则、故意忽略等方式绕过。

开发中要多转换调试思路，尝试不同的操作。

如果5次尝试小修小补都没写对，不要反复尝试，直接读完原文件所有内容，然后备份，整体重写，对照原逻辑直到写对为止。

如果持续没写对，应该用搜素引擎对关键信息进行深度搜索，而不是反复在错误的道路上尝试，对搜索到的信息必须交叉验证是否可靠。

工程检查或审计要点：

1. 功能完备性
2. 是否存在线程竞争、死锁、内存泄漏、
3. 始终实证当前设计方案为最优

# 高风险代码错误检查清单

## 1. 线程竞争 / 数据竞争

* **问题描述**：多个线程、协程、异步任务或进程同时读写共享状态，且缺乏正确同步。
* **典型表现**：

  * 共享变量被并发修改。
  * 读写顺序依赖调度时机。
  * 偶发性错误，难以稳定复现。
  * 单测通过，但高并发或生产环境异常。
* **重点检查**：

  * 是否存在共享可变状态。
  * 共享状态是否由锁、原子变量、消息队列、不可变对象或事务机制保护。
  * 复合操作是否被误认为是原子的，例如“读取-修改-写回”。
  * 缓存、计数器、状态机、连接池、任务队列是否线程安全。
* **避免方式**：

  * 优先减少共享可变状态。
  * 对共享状态使用明确同步机制。
  * 使用线程安全容器时，仍需检查复合操作是否安全。
  * 对并发代码增加压力测试和竞态测试。

## 2. 死锁

* **问题描述**：多个执行单元互相等待对方释放资源，导致程序永久阻塞。
* **典型表现**：

  * 线程卡死，CPU 占用不高。
  * 请求无响应但进程仍存活。
  * 日志停在获取锁、等待队列、等待连接等位置。
* **重点检查**：

  * 是否存在多个锁嵌套获取。
  * 不同代码路径的加锁顺序是否一致。
  * 是否在持锁状态下执行阻塞 I/O、RPC、数据库操作、回调或等待异步任务。
  * 是否存在锁未释放、异常路径未解锁。
* **避免方式**：

  * 统一锁获取顺序。
  * 缩小锁粒度和持锁时间。
  * 避免持锁调用外部代码。
  * 使用超时锁、try-lock 或死锁检测机制。
  * 使用 RAII、defer、finally 等机制保证释放。

## 3. 活锁与饥饿

* **问题描述**：程序没有完全阻塞，但任务长期无法推进。
* **典型表现**：

  * 线程持续运行但没有有效进展。
  * 某些任务长期得不到执行机会。
  * 重试、让步、抢占机制反复触发。
* **重点检查**：

  * 是否存在无限重试。
  * 是否存在优先级过高任务长期占用资源。
  * 是否存在公平性不足的锁、队列或调度策略。
  * 是否存在忙等、自旋等待或无退避重试。
* **避免方式**：

  * 为重试增加退避、抖动和最大次数。
  * 为等待设置超时。
  * 检查任务队列是否存在饥饿风险。
  * 避免无条件 while-loop 轮询。

## 4. 内存泄漏

* **问题描述**：程序不再需要的内存仍然无法释放，导致内存占用持续增长。
* **典型表现**：

  * 进程内存随运行时间持续上涨。
  * 请求完成后对象仍然存活。
  * 长时间运行后出现 OOM。
* **重点检查**：

  * C/C++ 中是否存在 malloc/new 后未释放。
  * 是否存在异常路径、提前 return 路径未释放内存。
  * 是否存在容器、缓存、全局对象长期持有引用。
  * 是否存在监听器、回调、订阅关系未注销。
  * 是否存在 ThreadLocal、协程上下文、任务上下文未清理。
* **避免方式**：

  * C/C++ 优先使用 RAII、智能指针和所有权约束。
  * 托管语言中重点检查对象是否被 GC Root 长期持有。
  * 为缓存设置容量上限、过期策略和淘汰策略。
  * 为监听器、订阅、回调提供明确注销路径。
  * 使用内存分析工具检查对象保留链。

## 5. GC 语言中的长期可达对象

* **问题描述**：Java、C#、Go 等语言虽然有 GC，但对象只要仍可从 GC Root 访问，就不会被回收。
* **典型表现**：

  * Full GC 频繁但内存无法明显下降。
  * 老年代、堆、goroutine、线程上下文持续增长。
  * 缓存、静态 Map、单例对象持有大量历史数据。
* **重点检查**：

  * 是否存在静态集合、全局缓存、单例对象长期持有引用。
  * 是否存在 ThreadLocal 未 remove。
  * 是否存在事件监听器未注销。
  * 是否存在闭包捕获大对象。
  * 是否存在队列堆积导致历史任务对象无法释放。
* **避免方式**：

  * 不把短生命周期对象挂到长生命周期对象上。
  * 清理 ThreadLocal、监听器、订阅、回调。
  * 控制缓存容量。
  * 检查对象引用链，而不是只检查是否存在循环引用。
* **注意**：

  * 单纯的对象循环引用通常不是 GC 语言中的泄漏原因。
  * 真正需要检查的是对象是否仍然从 GC Root 可达。

## 6. 悬垂指针 / Use-After-Free

* **问题描述**：对象或内存已经释放，但代码仍继续访问它。
* **典型表现**：

  * 随机崩溃。
  * 数据被异常修改。
  * 同一段代码有时正常，有时异常。
  * 安全漏洞风险极高。
* **重点检查**：

  * 指针释放后是否仍被使用。
  * 对象所有权是否清晰。
  * 容器扩容、对象移动后，旧指针或引用是否仍被使用。
  * 异步任务是否引用了已经销毁的对象。
* **避免方式**：

  * 明确对象所有权。
  * 避免裸指针跨生命周期传递。
  * 使用智能指针、生命周期标记或引用计数。
  * 释放后立即清空指针并避免继续传播。

## 7. 重复释放

* **问题描述**：同一块资源被释放两次或多次。
* **典型表现**：

  * 程序崩溃。
  * 堆结构损坏。
  * 偶发性异常。
* **重点检查**：

  * 多个对象是否共同认为自己拥有同一资源。
  * 异常路径和正常路径是否都会释放同一资源。
  * close/free/delete 是否可能重复执行。
* **避免方式**：

  * 使用单一所有权模型。
  * 使用 RAII、defer、finally 管理释放。
  * 释放操作设计为幂等时，需要明确记录状态。

## 8. 数组越界 / 切片越界

* **问题描述**：访问数组、切片、字符串、缓冲区时，下标超过合法范围。
* **典型表现**：

  * 崩溃。
  * 数据损坏。
  * 安全漏洞。
  * 托管语言中抛出越界异常。
* **重点检查**：

  * 下标是否可能等于长度。
  * 循环边界是否正确。
  * 空数组、单元素数组是否覆盖。
  * 分页、批处理、窗口滑动、字符串截取是否存在 off-by-one。
  * 长度单位是否一致，例如字节数、字符数、元素数。
* **避免方式**：

  * 明确使用半开区间 `[start, end)`。
  * 对外部输入的长度和下标做校验。
  * 增加边界用例测试：空、1、最大长度、刚好越界。
  * 避免手写复杂下标计算。

## 9. Off-by-One 错误

* **问题描述**：边界条件差一位，导致遗漏一个元素或多处理一个元素。
* **典型表现**：

  * 第一条或最后一条数据处理异常。
  * 分页结果重复或遗漏。
  * 时间区间多算或少算一天。
* **重点检查**：

  * 循环条件是 `<` 还是 `<=`。
  * 区间是闭区间、开区间还是半开区间。
  * 分页 offset、limit、pageNo、pageSize 的定义是否一致。
  * 时间区间是否包含起止时刻。
* **避免方式**：

  * 统一区间约定，优先使用半开区间。
  * 明确命名变量，例如 `startInclusive`、`endExclusive`。
  * 使用边界测试覆盖首尾元素。

## 10. 空指针 / 空引用

* **问题描述**：变量、对象、集合、配置或返回值为空，但代码直接访问。
* **典型表现**：

  * NullPointerException。
  * panic。
  * segmentation fault。
  * 生产环境由罕见输入触发。
* **重点检查**：

  * 外部输入是否可能为空。
  * 查询结果是否可能不存在。
  * 配置项是否可能缺失。
  * map/list/object 字段是否可能为空。
  * 异常路径返回值是否与正常路径一致。
* **避免方式**：

  * 明确函数是否允许返回空。
  * 使用 Optional、Result、错误码或显式异常。
  * 对外部输入和依赖返回值做防御性检查。
  * 避免把空值当作正常业务状态隐藏。

## 11. 未初始化变量

* **问题描述**：变量、结构体字段、对象状态未被正确初始化就被使用。
* **典型表现**：

  * 随机值。
  * 状态机进入非法状态。
  * 某些分支下才出现错误。
* **重点检查**：

  * 构造函数是否覆盖所有字段。
  * 条件分支是否都有赋值。
  * 默认值是否具有明确业务含义。
  * 配置加载失败后是否继续使用默认值。
* **避免方式**：

  * 初始化时强制提供必要字段。
  * 使用不可变对象或 builder 校验。
  * 禁止无意义默认值。
  * 对状态机增加非法状态检查。

## 12. 整数溢出

* **问题描述**：整数运算超过类型范围，导致结果回绕、截断或变成负数。
* **典型表现**：

  * 容量计算错误。
  * 下标变成负数。
  * 金额、长度、时间戳异常。
  * 安全漏洞。
* **重点检查**：

  * 长度、容量、offset、size 是否由乘法或加法计算。
  * 外部输入是否可能导致极大数值。
  * 类型转换是否从大范围转小范围。
  * 有符号和无符号整数是否混用。
* **避免方式**：

  * 对输入范围做上限校验。
  * 使用安全整数运算。
  * 检查类型转换。
  * 对容量计算增加溢出保护。

## 13. 浮点精度错误

* **问题描述**：使用浮点数处理需要精确计算的值，导致误差累积。
* **典型表现**：

  * 金额计算不一致。
  * 比较结果异常。
  * 汇总值和明细值对不上。
* **重点检查**：

  * 金额、费率、比例、评分是否使用了不合适的浮点类型。
  * 是否直接比较浮点数相等。
  * 是否存在多次累加误差。
* **避免方式**：

  * 金额优先使用整数分、定点数或 Decimal。
  * 浮点比较使用误差阈值。
  * 明确舍入规则。

## 14. 除零错误

* **问题描述**：分母可能为 0，但代码未检查。
* **典型表现**：

  * 程序崩溃。
  * 返回 NaN、Infinity。
  * 指标计算异常。
* **重点检查**：

  * 比率、均值、命中率、增长率、转化率是否可能分母为 0。
  * 空集合统计是否特殊处理。
  * 历史值为 0 时是否仍计算同比、环比。
* **避免方式**：

  * 明确分母为 0 的业务语义。
  * 对指标计算统一封装安全除法。
  * 在报表中区分 0、空值和不可计算。

## 15. 资源泄漏

* **问题描述**：非内存资源未释放，例如文件、socket、数据库连接、锁、线程、协程、GPU 资源等。
* **典型表现**：

  * 文件句柄耗尽。
  * 连接池耗尽。
  * 请求阻塞。
  * 显存、线程数或协程数持续增长。
* **重点检查**：

  * 文件、连接、事务、游标是否关闭。
  * 锁是否释放。
  * 线程池、定时任务、协程是否停止。
  * GPU tensor、上下文、缓存是否释放。
  * 异常路径是否覆盖资源释放。
* **避免方式**：

  * 使用 try-with-resources、defer、finally、RAII。
  * 给连接和请求设置超时。
  * 统一资源所有权和释放责任。
  * 增加资源监控指标。

## 16. 异常路径资源未释放

* **问题描述**：主流程能正确释放资源，但异常、提前返回、取消任务时没有释放。
* **典型表现**：

  * 偶发资源泄漏。
  * 错误请求后系统状态异常。
  * 重试后副作用叠加。
* **重点检查**：

  * 每个 return、throw、panic、break、continue 路径是否释放资源。
  * 异步取消是否触发清理逻辑。
  * 事务失败后是否回滚。
  * 临时文件是否删除。
* **避免方式**：

  * 将释放逻辑放入 finally、defer、RAII。
  * 对失败路径单独写测试。
  * 避免在多个分支手工重复释放。

## 17. 事务边界错误

* **问题描述**：事务范围过大、过小或与业务一致性边界不匹配。
* **典型表现**：

  * 部分数据写入成功。
  * 数据库与外部系统状态不一致。
  * 长事务导致锁等待。
* **重点检查**：

  * 多表写入是否在同一事务内。
  * 事务内是否包含远程调用、长时间计算或阻塞 I/O。
  * 失败时是否回滚。
  * 提交后事件、消息、缓存更新是否可靠。
* **避免方式**：

  * 明确事务保护的数据范围。
  * 避免长事务。
  * 使用 outbox、消息表、幂等消费等模式处理跨系统一致性。
  * 明确补偿流程。

## 18. 丢失更新

* **问题描述**：多个请求基于旧值同时更新，后写入覆盖先写入。
* **典型表现**：

  * 计数不准确。
  * 库存、余额、状态被覆盖。
  * 并发下偶发数据错误。
* **重点检查**：

  * 是否存在先查再改再写。
  * 是否缺少版本号、CAS、行锁或原子更新。
  * 缓存回写是否覆盖数据库新值。
* **避免方式**：

  * 使用乐观锁、悲观锁或原子 SQL。
  * 对状态流转增加版本检查。
  * 对关键写入增加并发测试。

## 19. 重复提交 / 非幂等重试

* **问题描述**：请求失败后重试，导致重复写入、重复扣款、重复发送、重复创建。
* **典型表现**：

  * 重复订单。
  * 重复扣费。
  * 重复消息。
  * 重复任务执行。
* **重点检查**：

  * 接口是否支持幂等键。
  * 重试逻辑是否区分可重试和不可重试错误。
  * 超时后实际服务端是否可能已经成功。
  * 消息消费是否可能重复投递。
* **避免方式**：

  * 为写接口设计幂等键。
  * 对外部调用记录请求状态。
  * 消息消费使用去重表或唯一约束。
  * 明确重试次数和退避策略。

## 20. 缓存一致性错误

* **问题描述**：缓存与数据库、文件、远端状态不一致。
* **典型表现**：

  * 用户看到旧数据。
  * 更新后查询结果不稳定。
  * 并发更新后缓存覆盖新值。
* **重点检查**：

  * 更新数据库后是否清理或更新缓存。
  * 缓存过期时间是否合理。
  * 是否存在缓存击穿、穿透、雪崩。
  * 并发重建缓存是否可能写入旧值。
* **避免方式**：

  * 明确缓存失效策略。
  * 对热点 key 增加互斥重建。
  * 使用版本号或时间戳避免旧值覆盖新值。
  * 对缓存和数据库差异增加监控。

## 21. 无界队列 / 无界缓存

* **问题描述**：队列、缓存、任务池没有容量限制，流量升高时资源持续增长。
* **典型表现**：

  * 内存持续上涨。
  * 延迟不断变大。
  * 系统最终 OOM。
* **重点检查**：

  * 队列是否有最大长度。
  * 缓存是否有容量上限。
  * 生产速度是否可能超过消费速度。
  * 积压后是否有降级、拒绝或丢弃策略。
* **避免方式**：

  * 设置容量上限。
  * 使用背压机制。
  * 增加超时、限流、熔断。
  * 监控队列长度和消费延迟。

## 22. 阻塞调用导致线程池耗尽

* **问题描述**：有限线程池内执行长时间阻塞操作，导致任务堆积。
* **典型表现**：

  * 请求大量超时。
  * 线程数达到上限。
  * 队列持续增长。
* **重点检查**：

  * 线程池中是否执行阻塞 I/O。
  * 是否在异步任务中同步等待另一个异步任务。
  * 是否存在嵌套提交任务并等待结果。
  * 数据库、RPC、HTTP 调用是否设置超时。
* **避免方式**：

  * 区分 CPU 线程池和 I/O 线程池。
  * 避免在线程池中阻塞等待同一线程池任务。
  * 所有外部调用设置超时。
  * 对线程池队列设置上限。

## 23. 异步任务取消处理错误

* **问题描述**：任务取消后，资源未释放或状态未回滚。
* **典型表现**：

  * 用户取消请求后后台仍继续执行。
  * 临时状态残留。
  * 资源持续占用。
* **重点检查**：

  * 是否传递取消信号。
  * 取消后是否释放锁、连接、文件、GPU 资源。
  * 部分结果是否被错误提交。
  * 父任务取消是否传播到子任务。
* **避免方式**：

  * 统一取消机制。
  * 在取消路径执行清理逻辑。
  * 对异步任务设置超时。
  * 避免 fire-and-forget 任务无监控运行。

## 24. 状态机非法状态

* **问题描述**：对象或流程进入未定义状态，导致后续逻辑异常。
* **典型表现**：

  * 状态跳转不符合业务规则。
  * 同一对象同时具有互斥状态。
  * 异常路径后状态无法恢复。
* **重点检查**：

  * 状态枚举是否完整。
  * 状态转移是否集中管理。
  * 是否允许非法跳转。
  * 并发状态更新是否受保护。
* **避免方式**：

  * 明确状态转移表。
  * 对非法状态直接失败。
  * 状态更新使用事务或锁。
  * 增加状态转移单测。

## 25. 输入未校验

* **问题描述**：外部输入未经校验直接进入业务逻辑、数据库、文件系统或命令执行。
* **典型表现**：

  * 异常数据导致崩溃。
  * 安全漏洞。
  * 数据污染。
* **重点检查**：

  * 参数类型、长度、范围、格式是否校验。
  * 枚举值是否白名单限制。
  * 文件名、路径、URL 是否校验。
  * JSON、XML、表单字段是否允许多余字段。
* **避免方式**：

  * 所有外部输入默认不可信。
  * 使用白名单而非黑名单。
  * 在边界层统一校验。
  * 对异常输入返回明确错误。

## 26. 注入风险

* **问题描述**：用户输入被拼接到 SQL、命令、脚本、模板、表达式或路径中执行。
* **典型表现**：

  * SQL 注入。
  * 命令注入。
  * 模板注入。
  * 表达式注入。
* **重点检查**：

  * 是否拼接 SQL。
  * 是否拼接 shell 命令。
  * 是否动态执行脚本或表达式。
  * 是否把用户输入传给模板引擎。
* **避免方式**：

  * SQL 使用参数化查询。
  * 命令执行使用参数数组，避免 shell 拼接。
  * 动态表达式使用白名单。
  * 对模板上下文做转义和隔离。

## 27. 路径穿越

* **问题描述**：用户通过 `../`、绝对路径、软链接等方式访问预期目录外的文件。
* **典型表现**：

  * 任意文件读取。
  * 任意文件覆盖。
  * 配置或密钥泄露。
* **重点检查**：

  * 文件名是否来自用户输入。
  * 是否校验最终规范化路径。
  * 是否处理软链接。
  * 上传、下载、解压逻辑是否限制目录。
* **避免方式**：

  * 使用固定根目录。
  * 规范化路径后检查是否仍在根目录内。
  * 禁止直接使用用户提供的完整路径。
  * 解压文件时检查每个条目的目标路径。

## 28. 权限校验遗漏

* **问题描述**：接口只校验登录状态，但未校验资源访问权限。
* **典型表现**：

  * 用户访问他人数据。
  * 横向越权。
  * 管理接口被普通用户调用。
* **重点检查**：

  * 是否校验资源归属。
  * 是否区分读、写、删除、审批等权限。
  * 后端是否独立校验权限，而不是依赖前端隐藏按钮。
  * 批量接口是否逐条校验权限。
* **避免方式**：

  * 在服务端统一鉴权。
  * 对每个资源操作校验主体、动作、资源。
  * 对批量操作逐项检查。
  * 增加越权测试。

## 29. 敏感信息泄露

* **问题描述**：日志、错误信息、接口返回、缓存或监控中暴露敏感数据。
* **典型表现**：

  * 日志中出现密码、token、身份证号、手机号、密钥。
  * 异常栈返回给用户。
  * 调试接口暴露内部配置。
* **重点检查**：

  * 日志是否打印完整请求、响应、Header、Cookie。
  * 异常信息是否直接返回前端。
  * 配置、环境变量、密钥是否可能被输出。
  * 脱敏规则是否覆盖所有敏感字段。
* **避免方式**：

  * 统一日志脱敏。
  * 错误返回使用通用信息。
  * 密钥不进入代码仓库和日志。
  * 对调试接口增加访问控制。

## 30. 反序列化风险

* **问题描述**：反序列化不可信数据时触发任意对象构造、代码执行或资源消耗。
* **典型表现**：

  * 远程代码执行。
  * 内存或 CPU 被异常消耗。
  * 对象结构绕过校验。
* **重点检查**：

  * 是否反序列化用户可控数据。
  * 是否允许任意类型反序列化。
  * 是否存在多态类型自动绑定。
  * 是否限制输入大小和嵌套深度。
* **避免方式**：

  * 避免反序列化不可信对象格式。
  * 使用白名单类型。
  * 限制输入大小、深度和字段。
  * 优先使用简单数据格式和显式映射。

## 31. 时间处理错误

* **问题描述**：时区、夏令时、时间精度、时间区间处理不正确。
* **典型表现**：

  * 跨天统计错误。
  * 定时任务提前或延后。
  * 时间范围查询遗漏数据。
* **重点检查**：

  * 是否明确时区。
  * 是否混用本地时间和 UTC。
  * 时间区间是否统一为半开区间。
  * 时间戳单位是秒、毫秒、微秒还是纳秒。
  * 日期和时间是否被错误截断。
* **避免方式**：

  * 存储层优先使用 UTC。
  * 展示层再转换本地时区。
  * 统一时间单位。
  * 使用 `[start, end)` 表示时间范围。

## 32. 配置错误默认值

* **问题描述**：配置缺失或解析失败后，程序使用危险默认值继续运行。
* **典型表现**：

  * 安全开关失效。
  * 超时时间过长或没有超时。
  * 连接到错误环境。
* **重点检查**：

  * 必填配置缺失时是否直接失败。
  * 默认值是否安全。
  * 测试环境配置是否可能进入生产。
  * 配置热更新是否校验。
* **避免方式**：

  * 必填配置缺失时启动失败。
  * 默认值采用保守策略。
  * 启动时打印关键配置摘要，但避免泄露密钥。
  * 配置变更前做校验。

## 33. 错误处理不完整

* **问题描述**：错误被忽略、吞掉、错误转换丢失上下文，或继续执行导致更严重问题。
* **典型表现**：

  * 日志无异常但结果错误。
  * 上层收到成功状态但实际失败。
  * 根因难以定位。
* **重点检查**：

  * 返回错误是否被检查。
  * catch 块是否只打印日志但继续执行。
  * 错误码和异常是否丢失上下文。
  * 批处理部分失败是否有明确处理。
* **避免方式**：

  * 所有错误必须处理、传播或明确忽略。
  * 明确区分可恢复和不可恢复错误。
  * 保留错误上下文。
  * 对批处理定义部分成功策略。

## 34. 重试风暴

* **问题描述**：大量请求失败后同时重试，进一步放大故障。
* **典型表现**：

  * 下游故障被上游重试放大。
  * 流量尖峰。
  * 服务恢复困难。
* **重点检查**：

  * 是否有无限重试。
  * 是否没有退避和抖动。
  * 是否所有客户端同时重试。
  * 是否对不可重试错误也重试。
* **避免方式**：

  * 设置最大重试次数。
  * 使用指数退避和随机抖动。
  * 配合限流、熔断、降级。
  * 区分错误类型。

## 35. 日志不足或过量

* **问题描述**：日志太少无法定位问题，或日志过多影响性能、泄露数据。
* **典型表现**：

  * 出错时没有关键上下文。
  * 高并发下日志占用大量 I/O。
  * 日志中包含敏感信息。
* **重点检查**：

  * 关键失败路径是否有日志。
  * 日志是否包含请求 ID、任务 ID、资源 ID。
  * 是否在循环、高频路径打印大量日志。
  * 是否对敏感字段脱敏。
* **避免方式**：

  * 关键路径保留结构化日志。
  * 高频日志采样或降级。
  * 日志分级清晰。
  * 统一脱敏策略。

## 36. 测试只覆盖主流程

* **问题描述**：代码只验证正常路径，未覆盖边界、异常、并发和失败路径。
* **典型表现**：

  * 单测通过，生产异常。
  * 极端输入触发错误。
  * 外部依赖失败时行为不可控。
* **重点检查**：

  * 是否测试空输入、极大输入、非法输入。
  * 是否测试异常路径。
  * 是否测试并发访问。
  * 是否测试重试、超时、取消。
  * 是否测试资源释放。
* **避免方式**：

  * 每个高风险模块增加边界测试。
  * 为失败路径写测试。
  * 对并发逻辑做压力测试。
  * 使用故障注入验证外部依赖失败场景。

# 总体检查原则

* 涉及共享状态的代码，必须检查并发安全。
* 涉及锁、连接、文件、线程、协程、GPU 资源的代码，必须检查释放路径。
* 涉及数组、切片、字符串、分页、时间窗口的代码，必须检查边界。
* 涉及外部输入的代码，必须检查校验、权限和注入风险。
* 涉及数据库、缓存、消息队列、外部接口的代码，必须检查一致性和幂等性。
* 涉及异常、取消、超时、重试的代码，必须检查失败路径。
* 不能只看主流程是否跑通，必须检查边界条件、异常路径和并发场景。
